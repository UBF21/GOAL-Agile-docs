# Retrospectiva Basada en Datos

La Retrospectiva Basada en Datos es el evento de mejora del ciclo de GOAL. Se celebra inmediatamente después de la Goal Review y utiliza las métricas de flujo como input principal en lugar de opiniones del equipo.

---

## Descripción General del Evento

| Elemento | Detalle |
|---------|---------|
| Duración | 45–60 minutos |
| Facilitador | Flow Master |
| Participantes | Todos los roles del equipo |
| Frecuencia | Al final de cada Ciclo de Objetivos, después de la Goal Review |
| Input obligatorio | Flow Metrics Dashboard preparado antes de la reunión |

---

## Qué la Hace Basada en Datos

El modificador "basado en datos" no es decorativo — es una regla. Las discusiones de la retrospectiva deben estar ancladas a datos. Las declaraciones sin datos son redirigidas:

> "Siento que fuimos lentos este ciclo" se convierte en → "El Cycle Time fue 4,2 días vs 2,8 días el ciclo pasado"

Esto previene que la retrospectiva degenere en debates subjetivos, reduce el sesgo de recencia, y asegura que las mejoras apunten a problemas medidos reales.

---

## Agenda (4 Pasos)

### Paso 1 — Revisión de Métricas de Flujo (15 min)

Revisar el Flow Metrics Dashboard para el ciclo:
- Flow Efficiency: ¿dentro del benchmark? ¿tendencia hacia arriba o abajo?
- Block Rate: ¿dentro del benchmark? ¿qué tipos de bloqueantes fueron más comunes?
- Cycle Time: ¿estable? ¿aumentando?
- Delivery Rate: ¿consistente con ciclos anteriores?
- Focus Ratio: ¿Se excedió el límite de WIP en algún momento?

### Paso 2 — Análisis del Block Register (15 min)

Revisar todas las entradas del Block Register del ciclo:
- ¿Cuántos bloqueantes ocurrieron? ¿De qué tipos?
- ¿Cuánto tiempo tomó resolverlos?
- ¿Algún patrón recurrente?
- ¿Algún bloqueante del que se podría haber prevenido la recurrencia con un cambio de proceso?

### Paso 3 — Identificar Mejoras (20 min)

Basándose en las discusiones de las métricas y el Block Register, el equipo identifica 1–3 mejoras específicas con:
- Descripción de la mejora
- Propietario único (una persona responsable)
- Métrica de éxito (¿cómo sabremos si funcionó?)
- Criterio de revisión (¿cuándo lo revisaremos en la próxima retrospectiva?)

### Paso 4 — Revisar Mejoras del Ciclo Anterior (5 min)

Para cada mejora identificada en la retrospectiva anterior:
- ¿Se implementó?
- ¿Tuvieron efecto las métricas?

---

## Benchmarks de Referencia

### Eficiencia del Flujo (Flow Efficiency)

| Rango | Interpretación |
|-------|----------------|
| < 15% | La mayor parte del tiempo se pasa esperando; investigar causas |
| 15–30% | Típico para equipos nuevos en gestión de flujo |
| 30–50% | Saludable para equipos establecidos |
| 50%+ | Flujo de alto rendimiento |

### Tasa de Bloqueo (Block Rate)

| Rango | Interpretación |
|-------|----------------|
| < 10% | Excelente — flujo de trabajo muy limpio |
| 10–20% | Saludable para la mayoría de los equipos |
| 20–30% | Investigar patrones recurrentes |
| > 30% | Problema sistémico del proceso — requiere investigación profunda |

---

## Modos de Fallo

**Retrospectivas sin datos:** El Flow Metrics Dashboard no se preparó antes de la sesión. La solución: preparar el dashboard antes de la retrospectiva, no durante.

**Listas de mejoras largas:** La identificación de más de 3 mejoras casi garantiza que ninguna se implementará. Forzar una priorización.

**Mejoras sin propietario:** "El equipo" no es un propietario. Asignar una persona responsable para cada mejora.

**Sin revisión de mejoras anteriores:** Si las mejoras del ciclo anterior no se revisan, no hay accountability. Comenzar siempre con el Paso 4.
