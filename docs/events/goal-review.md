# Goal Review

**Purpose:** Evaluate whether the cycle achieved its objectives and measure the value delivered.

**Duration:** 30–45 minutes

**Participants:** All roles, plus any relevant stakeholders

**When it happens:** At the end of every Goal Cycle, before the retrospective

---

## Agenda

### [15–20 min] Step 1 — Review Each Primary Goal

For each primary goal defined in the Goal Cycle Plan:

1. **Was it completed?** — yes/no based on full DoD compliance
2. **Does it meet the Definition of Done?** — no partial credit; every DoD criterion must be fully met
3. **What is the evidence of value delivered?** — the Product Strategist presents concrete evidence using the 3-Layer Value Framework

**Important:** A goal is not "completed" unless it fully meets the Definition of Done. Not 90% done. Not "done except for the tests." Fully done.

---

### [5–10 min] Step 2 — Review Secondary Goals

What was completed, what was deferred?

Secondary goals were never required for cycle success, so this section is informational — what additional value was captured if capacity allowed.

---

### [5 min] Step 3 — Cycle Accuracy Review

Flow Master presents:
- Estimated cycle duration vs actual cycle duration
- Record in the Cycle Accuracy Index
- If variance occurred, which extension reason category applies?

This data feeds directly into planning for the next cycle.

---

### [10 min] Step 4 — Value Delivered Summary

Product Strategist presents the value delivered using the 3-Layer Value Framework:

- **Layer 1 Business Value:** What KPIs were affected? (revenue, cost, churn, compliance)
- **Layer 2 User Value:** How did the user experience improve?
- **Layer 3 Technical Value:** How did the team's ability to deliver improve?

This presentation is for stakeholder visibility, not for team scoring. Stakeholders attending the Goal Review should leave with a clear understanding of what was delivered and what measurable impact it had.

---

## What Goal Review Is NOT

- A demo (demos can be included, but the review is about goal achievement, not feature showcase)
- A sprint retrospective (that happens after the Goal Review)
- A performance review of individuals
- A place to negotiate what "done" means retroactively

---

## Goal Review vs Scrum Sprint Review

| Dimension | Goal Review | Scrum Sprint Review |
|-----------|-------------|---------------------|
| Focus | Goal achievement + value delivered | Sprint output review |
| Success measure | Goals met per DoD | Sprint backlog items completed |
| Value reporting | 3-Layer Value Framework with evidence | Typically not structured |
| Cycle accuracy | Reviewed and logged in index | Not standard |
| Stakeholder role | Informed with evidence | Provide feedback |

---

## Handling Partially Completed Cycles

If not all primary goals were completed:

1. **Assess each incomplete goal** — what percentage was completed? What remains?
2. **Record in the Goal Cycle Plan** — partial completion with reason
3. **Evaluate for next cycle** — does the remaining work carry forward as a primary goal in the next cycle?
4. **Cycle Accuracy Index** — record as an extension or partial completion

Partial completion is not a failure if the root cause is understood and actioned in the retrospective.

---

## Value Evidence Requirements

The Product Strategist's value report is not subjective. Evidence requirements:

| Layer | Evidence types |
|-------|---------------|
| Business Value | Metric before and after (conversion rate, support tickets, revenue impact); compliance confirmation |
| User Value | User feedback, error rate change, task completion rate; A/B test data |
| Technical Value | Test coverage change, deployment frequency change, specific technical measurement |

If value cannot be measured at the time of the Goal Review (some value takes time to manifest), the Goal Review records an **expected measurement date** and the outcome is tracked in the following retrospective.
