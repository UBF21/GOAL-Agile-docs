# Program-Level Events

Three events operate at the program level. They coordinate across teams without replacing any single-team events.

---

## Program Sync

**Purpose:** Align all teams at the start of each cycle. Identify dependencies, confirm the Cross-Team Cycle Plan, and surface risks before work begins.

**Duration:** 30–45 minutes

**Participants:** Program Flow Master (facilitator), Program Strategist, all Product Strategists, all Flow Masters. Delivery Team members are optional unless a specific technical dependency requires their input.

**Cadence:** Once per cycle, at cycle start — same day as individual teams' Smart Planning Sessions, or the day before.

### Agenda

```
1. Program Goals review (5 min)
   - Program Strategist confirms which Program Goals are active this cycle
   - Any changes to program direction are communicated here
     (not in individual team planning)

2. Team cycle goal declarations (10 min)
   - Each Product Strategist states their team's primary goals for this cycle
   - No discussion — declarations only at this point

3. Dependency identification (10–15 min)
   - Program Flow Master asks: "Which goals or tasks in your team depend on
     another team's output?"
   - Each dependency is logged in the Dependency Register in real time
   - Providing teams confirm they can meet the expected delivery date
   - At-risk dependencies are flagged immediately

4. Critical path review (5 min)
   - Program Flow Master presents the preliminary critical path
   - Teams confirm sequence accuracy

5. Risk and mitigation (5 min)
   - Known risks to cross-team delivery
   - Mitigations or contingency plans for each
```

### What the Program Sync Is NOT

- A status report on what happened last cycle (that belongs in the Program Retrospective)
- A technical design session (those are bilateral between teams)
- A forum for scope negotiation (Program Strategist decisions are communicated beforehand)

## Cross-Team Dependency Review

**Purpose:** Maintain visibility on active dependencies mid-cycle. Surface at-risk items before they become blockers.

**Duration:** 15–20 minutes (hard limit)

**Participants:** Program Flow Master (facilitator), all Flow Masters. Product Strategists join only when a dependency is At Risk or Blocked.

**Cadence:** Twice per week during the cycle (e.g., Tuesday and Thursday).

### Agenda

```
1. Dependency Register review (10 min)
   - Walk through each Active dependency
   - Each providing team confirms: "On track / At Risk / Blocked"
   - At Risk dependencies: what is the mitigation?
   - Blocked dependencies: Program Flow Master takes ownership of escalation

2. New dependencies (5 min)
   - Any new cross-team dependencies identified since the last review?
   - Log and assign immediately

3. Resolved dependencies (2 min)
   - Confirm and close any dependencies resolved since last review
```

### Escalation Rule

Any dependency that is Blocked for more than 24 hours without a resolution path triggers mandatory Program Strategist involvement. If the block affects a primary goal, it triggers an evaluation of whether the affected team's cycle needs a formal extension.

## Program Retrospective

**Purpose:** Improve cross-team coordination using program-level metrics data.

**Duration:** 60–90 minutes

**Participants:** Program Flow Master (facilitator), Program Strategist, all Flow Masters, all Product Strategists. Optional: 1–2 Delivery Team members from each team.

**Cadence:** Once per program cycle — after all individual teams have run their Goal Reviews and team-level retrospectives.

### Agenda

```
1. Program metrics review (20 min)
   - Cross-Team Block Rate: what percentage of tasks encountered a cross-team blocker?
   - Average dependency resolution time: were dependencies resolved within 2 working days?
   - Dependency Register health: how many dependencies were identified, resolved, and blocked?
   - Program Goals completion: which Program Goals were fully delivered?

2. Dependency pattern analysis (15 min)
   - Which team pairs have the most dependencies?
   - Are any dependency types recurring (always the same type of work)?
   - Are there structural architectural changes that would reduce dependency count?

3. Coordination event effectiveness (10 min)
   - Were the Program Sync and Cross-Team Dependency Reviews useful?
   - Did the format need adjusting?

4. Improvement items (15 min)
   - 1–3 program-level improvements with owners and success metrics
   - Same rules as team-level retrospectives: maximum 3, all must have owners

5. Review previous improvements (10 min)
   - Were last program cycle's improvements implemented?
   - What do the metrics show?
```

---

## Program Event Cadence

| Event | Cadence | Duration | Who |
|-------|---------|----------|-----|
| Program Sync | Once per cycle (cycle start) | 30–45 min | PFM, PS, all Product Strategists, all Flow Masters |
| Cross-Team Dependency Review | Twice per week | 15–20 min | PFM, all Flow Masters |
| Program Retrospective | Once per program cycle (after team retros) | 60–90 min | PFM, PS, all roles |

*PFM = Program Flow Master, PS = Program Strategist*
